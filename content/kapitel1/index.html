    Kapitel 1 – Programmiersprachen im Überblick¶

    Was du in diesem Kapitel lernst¶

    • Was Programmiersprachen sind und wozu sie dienen
    • Wie Programmiersprachen nach Paradigmen unterschieden werden (imperativ, objektorientiert, funktional, skriptbasiert)
    • Welche typischen Einsatzszenarien es gibt
    • Was der Unterschied zwischen Compiler- und Interpreter-Sprachen ist

    So gehst du vor¶

    1. Lies die Kapitelinhalte und präge dir die Kernbegriffe ein.
    2. Bearbeite die Kurzübungen der Reihe nach – von Grundlagen bis Experte.
    3. Arbeite die Workshop-Aufgabe durch. Sie vertieft das Gelernte an einem zusammenhängenden Szenario.

    1.1 Was ist eine Programmiersprache?¶

    Eine Programmiersprache ist eine formale Sprache, mit der Menschen einem Computer Anweisungen erteilen können. Computer selbst verstehen nur Maschinencode – Folgen von Nullen und Einsen. Programmiersprachen dienen als Brücke: Der Mensch schreibt lesbaren Quellcode, der anschließend in Maschinencode übersetzt wird.

    Jede Programmiersprache besitzt eine Syntax (Grammatikregeln) und eine Semantik (Bedeutung der Anweisungen). Wer eine Programmiersprache beherrscht, kann damit Probleme in eine Abfolge präziser Anweisungen zerlegen – das nennt man algorithmisches Denken.

    Warum so viele Sprachen?

    Es gibt heute über 700 aktiv genutzte Programmiersprachen. Jede entstand, weil bestehende Sprachen für bestimmte Anwendungsfälle ungeeignet waren – zu langsam, zu komplex, zu fehleranfällig oder einfach nicht vorhanden.

    1.2 Programmierparadigmen¶

    Ein Paradigma ist eine grundlegende Denkweise, nach der ein Programm aufgebaut wird. Die wichtigsten Paradigmen im Überblick:

    Imperatives Paradigma¶

    Das Programm beschreibt wie etwas getan werden soll – Schritt für Schritt, in einer festen Reihenfolge. Der Programmablauf wird durch Anweisungen, Schleifen und Bedingungen gesteuert.

    Beispiele: C, Pascal, frühe Versionen von BASIC

    Typischer Einsatz: Systemnahe Programmierung, Mikrocontroller, Algorithmen

    Objektorientiertes Paradigma (OOP)¶

    Das Programm wird als Zusammenspiel von Objekten modelliert. Jedes Objekt hat Eigenschaften (Attribute) und Fähigkeiten (Methoden). Objekte kommunizieren miteinander über Nachrichten (Methodenaufrufe).

    Beispiele: Java, C#, Python, PHP (ab Version 5), Ruby

    Typischer Einsatz: Unternehmensanwendungen, Desktop-Software, Web-Backends

    Konzept Erklärung Beispiel
    Klasse Bauplan für Objekte Klasse Auto
    Objekt Konkretes Exemplar einer Klasse meinAuto
    Attribut Eigenschaft eines Objekts Farbe = rot
    Methode Funktion, die zu einem Objekt gehört fahren()
    Vererbung Eine Klasse übernimmt Eigenschaften einer anderen Elektroauto erbt von Auto

    Funktionales Paradigma¶

    Das Programm besteht aus mathematischen Funktionen, die Eingaben auf Ausgaben abbilden. Es gibt keine Zustände und keine Seiteneffekte – eine Funktion liefert bei gleicher Eingabe immer dasselbe Ergebnis.

    Beispiele: Haskell, Erlang, Elixir; funktionale Elemente auch in JavaScript, Python, Scala

    Typischer Einsatz: Datenverarbeitung, parallele Berechnungen, kritische Systeme

    Skriptbasiertes Paradigma¶

    Skriptsprachen sind darauf ausgelegt, bestehende Systeme zu steuern oder zu automatisieren. Sie werden meist nicht kompiliert, sondern direkt ausgeführt. Oft werden sie in Kombination mit anderen Sprachen eingesetzt.

    Beispiele: PHP, JavaScript, Python, Bash, PowerShell

    Typischer Einsatz: Webentwicklung, Systemautomatisierung, Dateiverarbeitung, Glue-Code

    Paradigmen schließen sich nicht aus

    Viele moderne Sprachen unterstützen mehrere Paradigmen. PHP z. B. ist skriptbasiert und gleichzeitig objektorientiert. Python wird imperativ, objektorientiert und funktional eingesetzt.

    1.3 Einsatzszenarien¶

    Welche Sprache wählt man für welches Projekt? Eine Orientierung:

    Einsatzbereich Typische Sprachen Begründung
    Webentwicklung (Frontend) JavaScript, TypeScript Laufen direkt im Browser
    Webentwicklung (Backend) PHP, Python, Java, Node.js Serverseitige Logik, Datenbankzugriff
    Mobile Apps Swift (iOS), Kotlin (Android), Flutter Native Performance oder plattformübergreifend
    Desktop-Software C#, Java, C++ Zugriff auf Betriebssystem-Ressourcen
    Datenwissenschaft / KI Python, R Umfangreiche Bibliotheken (NumPy, pandas, TensorFlow)
    Systemsoftware / Treiber C, C++, Rust Direkter Hardwarezugriff, maximale Performance
    Automatisierung / Scripting Bash, PowerShell, Python Schnelle Ausführung, kein Build-Schritt nötig
    Datenbanken SQL Deklarative Abfrage- und Manipulation von Daten

    1.4 Compiler vs. Interpreter¶

    Der wesentliche technische Unterschied zwischen Programmiersprachen liegt darin, wie Quellcode ausgeführt wird.

    Compiler-Sprachen¶

    Ein Compiler übersetzt den gesamten Quellcode vor der Ausführung in Maschinencode (oder Bytecode). Das Ergebnis ist eine ausführbare Datei, die ohne den Compiler läuft.

    flowchart LR
    A([Quellcode]) --> B[Compiler] --> C([Ausführbare Datei]) --> D([Ausführung])

    Vorteile: Sehr schnell in der Ausführung, Fehler werden vor dem Start erkannt
    Nachteile: Übersetzungsschritt nötig, für jede Plattform neu compilieren
    Beispiele: C, C++, Rust, Go, Java (zu Bytecode), C#

    Interpreter-Sprachen¶

    Ein Interpreter liest den Quellcode Zeile für Zeile und führt ihn direkt aus – ohne vorherige Kompilierung.

    flowchart LR
    A([Quellcode]) --> B[Interpreter] --> C([Direkte Ausführung\nZeile für Zeile])

    Vorteile: Keine Build-Phase, plattformunabhängig, leicht testbar
    Nachteile: Langsamer, Fehler treten erst zur Laufzeit auf
    Beispiele: PHP, Python, JavaScript (im Browser), Ruby, Bash

    Hybride Ansätze¶

    Manche Sprachen kombinieren beide Ansätze:

    • Java compiliert zu Bytecode, der dann von der Java Virtual Machine (JVM) interpretiert wird
    • Python compiliert intern zu .pyc-Bytecode, bevor der Interpreter diesen ausführt
    Merkmal Compiler Interpreter
    Übersetzungszeitpunkt Vor der Ausführung Zur Laufzeit
    Ausführungsgeschwindigkeit Schneller Langsamer
    Fehlerkennung Beim Kompilieren Erst bei Ausführung
    Typische Sprachen C, C++, Rust, Go PHP, Python, JavaScript
    Build-Schritt nötig? Ja Nein

    Fehlerzeitpunkt ist prüfungsrelevant

    Ein häufiger Prüfungspunkt: Bei Compiler-Sprachen werden Syntaxfehler vor der Ausführung erkannt. Bei Interpreter-Sprachen erst beim Ausführen der betreffenden Zeile.

    Kurzübungen¶

    Grundlagen Paradigmen zuordnen

    Ordne die folgenden Programmiersprachen dem passenden Paradigma zu.

    Programmiersprachen: Java, PHP, Haskell, C, JavaScript, Python

    Erstelle eine Tabelle mit den Spalten: - Sprache - Paradigma (imperativ / objektorientiert / funktional / skriptbasiert) - Typisches Einsatzszenario (1 Satz)

    Hinweis: Manche Sprachen unterstützen mehrere Paradigmen – wähle das dominante oder nenne beide.

    Tipp

    Denke daran: Skriptsprachen dienen oft zur Steuerung und Automatisierung. Funktionale Sprachen betonen Seiteneffektfreiheit. OOP dreht sich um Klassen und Objekte.

    Lösung

    Mögliche Lösung:

    Sprache Paradigma Einsatzszenario
    Java Objektorientiert Unternehmensanwendungen, Android-Apps
    PHP Skriptbasiert + objektorientiert Webentwicklung (Serverseitig)
    Haskell Funktional Wissenschaftliche Berechnungen, kritische Systeme
    C Imperativ Systemsoftware, Betriebssysteme, Treiber
    JavaScript Skriptbasiert + objektorientiert Webentwicklung (Frontend, auch Backend)
    Python Imperativ + OOP + funktional Datenwissenschaft, KI, Skripting
    Fortgeschritten Compiler oder Interpreter?

    Beantworte die folgenden Fragen zu Compiler- und Interpretersprachen:

    1. Ein Entwickler hat ein Python-Skript geschrieben. Er führt es aus – und erhält eine Fehlermeldung in Zeile 15. Die Zeilen 1–14 wurden bereits ausgeführt. Um welchen Sprachtyp handelt es sich? Begründe kurz.

    2. Ein anderer Entwickler hat ein Java-Programm geschrieben. Beim ersten Ausführungsversuch meldet das System: „Compilation error in line 8: missing semicolon". Das Programm wurde noch nicht gestartet. Um welchen Sprachtyp handelt es sich? Begründe kurz.

    3. Erkläre in 2–3 Sätzen: Warum sind Compiler-Sprachen oft schneller in der Ausführung als Interpreter-Sprachen?

    Tipp

    Der entscheidende Hinweis: Wann tritt der Fehler auf – vor oder während der Ausführung?

    Lösung

    1. Interpreter-Sprache (Python). Der Interpreter führt Zeile für Zeile aus und stoppt erst beim Fehler. Zeilen 1–14 wurden bereits ausgeführt, was typisch für Interpreter ist.

    2. Compiler-Sprache (Java). Der Compiler prüft den gesamten Code vor der Ausführung. Der Fehler tritt schon beim Kompilierungsvorgang auf – das Programm läuft noch gar nicht.

    3. Compiler übersetzen den Quellcode einmalig in optimierten Maschinencode, bevor das Programm startet. Interpreter müssen jede Zeile zur Laufzeit lesen, parsen und ausführen – dieser Overhead macht sie langsamer. Zudem können Compiler Optimierungen vornehmen, die zur Laufzeit nicht mehr möglich wären.

    Experte Sprachauswahl begründen

    Du bist IT-Berater. Für die folgenden Projekte soll eine geeignete Programmiersprache empfohlen werden. Begründe deine Wahl jeweils mit dem Paradigma, dem Einsatzszenario und mindestens einem konkreten Vorteil.

    Projekt A: Eine Webseite für eine kleine Bäckerei soll erstellt werden. Sie zeigt Produkte, Öffnungszeiten und ein Kontaktformular. Das Budget ist gering, der Entwickler hat wenig Erfahrung.

    Projekt B: Ein Unternehmen möchte eine komplexe Warenwirtschaftssoftware entwickeln, die über Jahre gepflegt und von mehreren Teams erweitert werden soll.

    Projekt C: Ein Datenwissenschaftler soll große Datensätze analysieren und maschinelle Lernmodelle trainieren.

    Projekt D: Ein Firmware-Entwickler soll die Software für einen eingebetteten Mikrocontroller schreiben. Der Speicher ist auf 64 KB begrenzt.

    Tipp

    Denke an: Einstiegsfreundlichkeit, Wartbarkeit bei großen Projekten, verfügbare Bibliotheken und Hardware-Nähe.

    Lösung

    Projekt A: PHP oder Python. Beide sind Einstiegsfreundlich, schnell einsetzbar und für serverseitige Webentwicklung optimiert. Kein Build-Schritt nötig, viele fertige CMS (WordPress = PHP).

    Projekt B: Java oder C#. Objektorientiert, streng typisiert, gut für große Teams und langlebige Codebasen. Compiler prüft Fehler früh; umfangreiche Frameworks verfügbar (Spring, .NET).

    Projekt C: Python. Umfangreichste Bibliotheken für Datenwissenschaft (NumPy, pandas, scikit-learn, TensorFlow). Schnelle Iteration durch Interpreter; interaktive Notebooks (Jupyter).

    Projekt D: C oder C++. Imperative, hardwarenahe Sprachen mit direktem Speicherzugriff. Sehr effizienter Maschinencode, geringer Overhead – ideal für ressourcenbeschränkte Systeme.

    Workshop¶

    Workshop Programmiersprachen – Gesamtanalyse

    Szenario: Du wirst als Junior-Berater in ein mittelständisches Unternehmen (200 Mitarbeiter) eingesetzt. Das Unternehmen möchte seine IT-Landschaft modernisieren und stellt dir folgende Situation vor:

    • Bestandssystem A: Eine Lagerverwaltungssoftware in COBOL aus den 1980er-Jahren. Wenige Mitarbeiter kennen die Sprache noch.
    • Geplantes Projekt B: Eine neue webbasierte Bestellplattform für Großkunden (Frontend + Backend + Datenbankanbindung).
    • Geplantes Projekt C: Automatisierung von wiederkehrenden Datenexporten (tägliche CSV-Extraktion aus der Datenbank).

    Aufgabe (in Gruppenarbeit oder allein):

    1. Recherchiere kurz: Was ist COBOL? Welchem Paradigma entspricht es?

    2. Für Projekt B: Empfehle eine Frontend-Sprache und eine Backend-Sprache. Begründe beide Entscheidungen je mit Paradigma und Einsatzszenario (ca. 3–4 Sätze je Sprache).

    3. Für Projekt C: Welche Sprache würdest du empfehlen? Begründe kurz.

    4. Diskutiere: Was sind die Risiken, wenn Bestandssystem A einfach weiterläuft und nicht modernisiert wird? Nenne mindestens 3 Risiken.

    5. Fasse deine Empfehlungen in einem kurzen „Technologie-Memo" (max. 15 Sätze) zusammen, das du dem Geschäftsführer übergeben könntest.

    Deliverables: Technologie-Memo (schriftlich oder als Stichpunktliste)

    Tipp

    Tipp für die Gruppenarbeit: Teilt die Aufgaben auf – eine Person übernimmt die COBOL-Recherche, andere formulieren die Empfehlungen, eine Person schreibt das Memo.

    Lösung

    Bewertungskriterien:

    1. COBOL = imperativ, prozedurale Geschäftslogik, Stapelverarbeitung – entstanden in den 1950/60ern.

    2. Frontend: JavaScript / TypeScript (Browserstandard, reaktive UIs). Backend: Java, PHP, Node.js oder Python – jeweils gut begründete Wahl akzeptabel.

    3. Projekt C: Python oder Bash/PowerShell – einfach, schnell zu schreiben, gute CSV/Datenbankbibliotheken.

    4. Risiken: Kein Personal mehr vorhanden (Wissensverlust), keine modernen Libraries/Security-Updates, schwere Integration mit neuen Systemen, steigende Wartungskosten.

    5. Memo: Strukturierte Empfehlung mit Begründungen – vollständige Punkte 1–4 abgedeckt.
