    Kapitel 8 – UML-Diagramme im Überblick¶

    Was du in diesem Kapitel lernst¶

    • Was UML ist und warum es als Standard existiert
    • Welche vier Diagrammarten zentral sind: Use-Case-, Aktivitäts-, Klassen- und Sequenzdiagramm
    • Wie du die Grundelemente jedes Diagrammtyps erkennst und liest
    • Wie du für eine konkrete Aufgabe die passende UML-Darstellungsform auswählst

    So gehst du vor¶

    1. Lies die Kapitelinhalte und achte auf die Symbole und deren Bedeutung.
    2. Bearbeite die Kurzübungen der Reihe nach – von Grundlagen bis Experte.
    3. Arbeite die Workshop-Aufgabe durch. Sie vertieft das Gelernte an einem zusammenhängenden Szenario.

    8.1 Was ist UML?¶

    UML (Unified Modeling Language) ist eine standardisierte grafische Sprache zur Modellierung von Software-Systemen. Sie wurde in den 1990er-Jahren von Grady Booch, Ivar Jacobson und James Rumbaugh entwickelt und ist heute ein ISO-Standard (ISO/IEC 19505).

    UML ist sprachunabhängig: Ein UML-Diagramm beschreibt, was ein System tut oder wie es strukturiert ist – unabhängig davon, ob es in Java, PHP, C# oder einer anderen Sprache implementiert wird.

    Warum UML? - Einheitliche Notation für Entwickler, Analysten und Manager - Kommunikation über Systemstrukturen ohne Quellcode - Dokumentation, die auch nach Jahren noch verständlich ist - Grundlage für automatische Code-Generierung (in fortgeschrittenen Werkzeugen)

    UML umfasst über 13 Diagrammtypen. In der Praxis und in IHK-Prüfungen sind vier besonders wichtig.

    8.2 Use-Case-Diagramm¶

    Das Use-Case-Diagramm (Anwendungsfalldiagramm) zeigt, wer (Akteur) das System wozu (Anwendungsfall) nutzt. Es beschreibt das System aus der Außenperspektive.

    Elemente¶

    Element Symbol Bedeutung
    Systemgrenze Rechteck Grenze des modellierten Systems
    Akteur Strichmännchen Person oder System, das mit dem System interagiert
    Anwendungsfall Ellipse Eine Funktion, die das System dem Akteur bietet
    Assoziation Linie Akteur nutzt einen Anwendungsfall
    Include gestrichelte Pfeillinie <<include>> AF enthält zwingend einen anderen AF
    Extend gestrichelte Pfeillinie <<extend>> AF erweitert optional einen anderen AF

    Beispiel: Bibliothekssystem¶

    flowchart LR
    Leser(["👤 Leser"])
    Bibliothekarin(["👤 Bibliothekarin"])

    subgraph Bibliothekssystem
        UC1(["Buch suchen"])
        UC2(["Buch ausleihen"])
        UC3(["Buch zurückgeben"])
        UC4(["Konto verwalten"])
        UC5(["Login"])
        UC2 -->|&lt;&lt;include&gt;&gt;| UC5
    end

    Leser --- UC1
    Leser --- UC2
    Leser --- UC3
    Leser --- UC4
    Bibliothekarin --- UC1
    Bibliothekarin --- UC3
    Bibliothekarin --- UC4

    Wann einsetzen? Anforderungsphase – um zu zeigen, welche Funktionen das System bietet und wer sie nutzt.

    8.3 Aktivitätsdiagramm¶

    Das Aktivitätsdiagramm beschreibt den Ablauf von Aktivitäten – ähnlich einem PAP, aber mächtiger. Es kann Parallelverarbeitung, Verantwortlichkeitsbereiche (Swimlanes) und komplexe Verzweigungen darstellen.

    Elemente¶

    Element Symbol Bedeutung
    Startknoten Ausgefüllter Kreis Beginn des Ablaufs
    Endknoten Kreis mit Ring Ende des Ablaufs
    Aktion Abgerundetes Rechteck Eine ausführbare Aktivität
    Entscheidungsknoten Raute Verzweigung (Bedingung)
    Zusammenführung Raute Mehrere Pfade laufen zusammen
    Gabelung Dicke waagrechte Linie Start paralleler Aktivitäten
    Verbindung Dicke waagrechte Linie Synchronisation paralleler Aktivitäten
    Swimlane Vertikal/horizontal geteilte Bereiche Verantwortlichkeit (wer führt aus?)

    Beispiel: Bestellprozess mit Swimlanes¶

    flowchart TD
    subgraph Kunde
        A([●]) --> B[Bestellung aufgeben]
        H([◎])
    end
    subgraph System
        C{Verfügbar?}
        D[Rechnung erstellen]
        E[Absage senden]
    end
    subgraph Lager
        F[Ware senden]
    end

    B --> C
    C -->|Ja| D
    C -->|Nein| E
    D --> F
    F --> H
    E --> H

    Wann einsetzen? Wenn Abläufe mit Verzweigungen, Parallelität oder Verantwortlichkeiten modelliert werden sollen.

    8.4 Klassendiagramm¶

    Das Klassendiagramm ist das zentrale Strukturdiagramm der OOP. Es zeigt, welche Klassen ein System enthält, welche Attribute und Methoden sie haben und wie sie miteinander in Beziehung stehen.

    Klassen-Notation¶

    classDiagram
    class Klasse {
        -attribut1 : Typ
        +attribut2 : Typ
        +methode1() Typ
        +methode2(x : Typ)
    }

    - = private, + = public, # = protected

    Sichtbarkeiten¶

    Zeichen Bedeutung
    + public – von überall zugreifbar
    - private – nur innerhalb der Klasse
    # protected – innerhalb der Klasse und Unterklassen

    Beziehungstypen¶

    Beziehung Symbol Bedeutung
    Assoziation Linie Klassen kennen sich
    Vererbung Pfeil mit offenem Dreieck „ist ein" – Unterklasse erbt von Oberklasse
    Komposition Ausgefüllte Raute + Linie „besteht aus" – Teil kann ohne Ganzes nicht existieren
    Aggregation Offene Raute + Linie „enthält" – Teil kann ohne Ganzes existieren
    Abhängigkeit Gestrichelter Pfeil Eine Klasse nutzt eine andere kurzfristig

    Beispiel: Bibliothekssystem¶

    classDiagram
    class Buch {
        -isbn : String
        -titel : String
        +getISBN() String
        +getTitel() String
    }
    class Ausleihe {
        -datum : Date
        -rueckgabe : Date
        +verlängern()
    }
    class Leser {
        -name : String
        -ausweisNr : Int
        +ausleihen()
    }
    Buch "1" --> "*" Ausleihe : wird ausgeliehen über
    Ausleihe "*" --> "1" Leser : gehört zu

    Wann einsetzen? Entwurfsphase – um die statische Struktur des Systems zu modellieren.

    8.5 Sequenzdiagramm¶

    Das Sequenzdiagramm zeigt, wie Objekte miteinander kommunizieren – in welcher Reihenfolge Nachrichten (Methodenaufrufe) ausgetauscht werden. Es modelliert das dynamische Verhalten bei einem konkreten Ablauf.

    Elemente¶

    Element Symbol Bedeutung
    Akteur / Objekt Rechteck oben, gestrichelte Linie nach unten Teilnehmer am Ablauf
    Lebenslinie Senkrechte gestrichelte Linie Zeitachse des Objekts
    Aktivierungsbalken Schmales Rechteck auf Lebenslinie Objekt ist aktiv
    Nachricht Pfeil Methodenaufruf oder Rückgabe
    Rückgabe Gestrichelter Pfeil Antwort auf eine Nachricht

    Beispiel: Login-Vorgang¶

    sequenceDiagram
    actor N as Nutzer
    participant B as Browser
    participant S as Server
    participant D as Datenbank

    N->>B: login()
    B->>S: POST /login
    S->>D: prüfeUser()
    D-->>S: User-Objekt
    S-->>B: 200 OK
    B-->>N: Willkommen

    Wann einsetzen? Wenn die zeitliche Reihenfolge von Interaktionen zwischen Objekten dokumentiert werden soll.

    8.6 Welches Diagramm wann?¶

    Frage / Situation Diagramm
    Welche Funktionen bietet das System? Wer nutzt sie? Use-Case-Diagramm
    Wie läuft ein Prozess ab? (mit Verzweigungen, Swimlanes) Aktivitätsdiagramm
    Welche Klassen gibt es? Wie sind sie strukturiert? Klassendiagramm
    Wie kommunizieren Objekte in einem konkreten Ablauf? Sequenzdiagramm

    Kurzübungen¶

    Grundlagen UML-Diagrammarten zuordnen

    Ordne die folgenden Fragen dem passenden UML-Diagramm zu:

    Diagramm-Optionen: Use-Case-Diagramm · Aktivitätsdiagramm · Klassendiagramm · Sequenzdiagramm

    Fragen: 1. Welche Klassen gibt es im System, welche Attribute haben sie und wie stehen sie zueinander? 2. Wer nutzt das System und welche Funktionen stehen den Nutzern zur Verfügung? 3. In welcher Reihenfolge rufen Objekte Methoden aufeinander auf? 4. Wie läuft ein Geschäftsprozess ab – mit Verzweigungen und wer ist jeweils verantwortlich? 5. Eine Fachabteilung möchte sehen, welche Aktionen das System dem Buchhalter ermöglicht. 6. Ein Entwickler möchte dokumentieren, wie Browser, Server und Datenbank beim Login miteinander kommunizieren.

    Tipp

    Merke: Use Case = Wer nutzt Was · Aktivität = Ablauf + Rollen · Klasse = Struktur · Sequenz = Zeitliche Kommunikation

    Lösung
    1. Klassendiagramm
    2. Use-Case-Diagramm
    3. Sequenzdiagramm
    4. Aktivitätsdiagramm
    5. Use-Case-Diagramm (Akteur = Buchhalter, Anwendungsfälle = Aktionen)
    6. Sequenzdiagramm (zeitliche Abfolge der Kommunikation zwischen Browser, Server, Datenbank)

├──────────────────┤         │ + verlängern()   │
│ + getISBN()      │         └────────┬─────────┘
│ + getTitel()     │                  │ *
└──────────────────┘                  │
                                      │ 1
                             ┌────────┴─────────┐
                             │    Leser          │
                             ├──────────────────┤
                             │ - name: String   │
                             │ - ausweisNr: Int │
                             ├──────────────────┤
                             │ + ausleihen()    │
                             │ + zurückgeben()  │
                             └──────────────────┘

    Aufgaben:

    1. Nenne alle drei Klassen und ihre Attribute.
    2. Was bedeutet das - vor den Attributen? Was würde + bedeuten?
    3. Was bedeutet die 1 und die * an den Verbindungslinien?
    4. Kann ein Buch mehreren Lesern gleichzeitig ausgeliehen sein? Begründe anhand des Diagramms.
    5. Welche Methode hat ein Leser-Objekt? Was könnte ausleihen() intern tun?
    Tipp

    Die Zahlen an den Linien sind Kardinalitäten – wie viele Objekte der einen Klasse mit wie vielen der anderen verknüpft sind.

    Lösung

    1. Buch: isbn, titel, autor. Ausleihe: ausleihDatum, rueckgabeDatum. Leser: name, ausweisNr.

    2. - = private (nur innerhalb der Klasse zugreifbar). + = public (von außen zugreifbar).

    3. 1 = genau ein Objekt. * = null oder mehr Objekte. Ein Buch kann mit mehreren Ausleihen verknüpft sein (1 Buch → * Ausleihen, z. B. über die Zeit). Ein Leser kann mehrere Ausleihen haben.

    4. Das Diagramm zeigt 1 Buch – * Ausleihen, was für Ausleihen über die Zeit gilt (Buchhistorie). Ob gleichzeitige Ausleihe möglich ist, hängt von der Implementierungslogik ab – das Diagramm lässt es offen.

    5. Methoden: ausleihen() und zurückgeben(). ausleihen() könnte intern ein neues Ausleihe-Objekt erzeugen, das Datum setzen und den Buchbestand aktualisieren.

    Betrachte das folgende Use-Case-Diagramm für ein Online-Banking-System:

    ┌────────────────────────────────────────────────────┐
│                Online-Banking-System               │
│                                                    │
│   ( Konto ansehen )                               │
│   ( Überweisung durchführen ) ──<<include>>──► ( Login )
│   ( Dauerauftrag einrichten ) ──<<include>>──► ( Login )
│   ( Kontoauszug herunterladen )                   │
│   ( Passwort ändern ) ──<<include>>──► ( Login )  │
│                                                    │
└────────────────────────────────────────────────────┘
      👤 Kunde          👤 Bank-Administrator

    Der Bank-Administrator hat Zugriff auf: Konto ansehen, Kontoauszug herunterladen. Der Kunde hat Zugriff auf alle Funktionen.

    Aufgaben:

    1. Erkläre in eigenen Worten, was <<include>> bedeutet.
    2. Warum ist Login als <<include>>-Beziehung und nicht als eigenständiger Anwendungsfall (ohne Pfeil) modelliert?
    3. Gibt es eine Funktion, die beide Akteure nutzen können? Welche?
    4. Ergänze: Welcher neue Anwendungsfall wäre sinnvoll – und für welchen Akteur?
    5. Was ist der Unterschied zwischen <<include>> und <<extend>>?
    Tipp

    <> = wird immer ausgeführt (verpflichtend). <> = wird nur unter bestimmten Bedingungen ausgeführt (optional).

    Lösung

    1. <<include>> bedeutet, dass der aufrufende Anwendungsfall zwingend den eingebundenen AF enthält. Immer wenn eine Überweisung stattfindet, muss der Nutzer eingeloggt sein (Login wird immer ausgeführt).

    2. Login ist keine eigenständige nützliche Funktion für den Nutzer – er bucht keine „Logins". Es ist eine Voraussetzung für andere Funktionen. Deshalb wird es als <<include>> modelliert.

    3. Konto ansehen und Kontoauszug herunterladen (beide Akteure haben Zugriff darauf).

    4. Beispiele: „Zwei-Faktor-Authentifizierung einrichten" (Kunde), „Kontosperrung veranlassen" (Administrator).

    5. <<include>> = immer ausgeführt, verpflichtend. <<extend>> = optional, nur unter bestimmten Bedingungen. Beispiel: „Tipp anzeigen" könnte bei „Login" als <<extend>> modelliert sein – wird nur beim ersten Login angezeigt.

    Workshop¶

    Workshop UML-Diagramme für ein Fallbeispiel

    Szenario: Eine Arztpraxis möchte ihre Terminverwaltung digitalisieren. Du wirst beauftragt, drei UML-Diagramme zu erstellen.

    Informationen: - Patienten können Termine online buchen, absagen oder verschieben - Ärzte und Sprechstundenhilfen können Termine einsehen und bestätigen - Sprechstundenhilfe legt neue Patienten an - Jeder Patient hat: Name, Geburtsdatum, Krankenkasse, Patientennummer - Jeder Termin hat: Datum, Uhrzeit, Arzt, Patient, Status (gebucht/bestätigt/abgesagt) - Beim Buchen eines Termins: Patient wählt Arzt → System prüft Verfügbarkeit → Bei Verfügbarkeit: Termin wird angelegt und Bestätigungs-E-Mail versendet → Bei Nicht-Verfügbarkeit: Alternativzeiten werden vorgeschlagen

    Aufgaben:

    1. Erstelle ein Use-Case-Diagramm (Textbeschreibung): Welche Akteure gibt es? Welche Anwendungsfälle? Gibt es <<include>>-Beziehungen?

    2. Erstelle ein Klassendiagramm (Textbeschreibung): Klassen, Attribute, Methoden und Beziehungen (mit Kardinalitäten).

    3. Beschreibe ein Sequenzdiagramm für den Buchungsvorgang: Wer kommuniziert mit wem? In welcher Reihenfolge?

    4. Welches Diagramm würde ein Auftraggeber zuerst sehen wollen? Begründe.

    Deliverables: Drei Diagrammbeschreibungen + Begründung

    Tipp

    Beginne mit dem Use-Case-Diagramm – es ist die Außenperspektive und zeigt, worum es geht. Dann Klassendiagramm für die Struktur, dann Sequenz für einen konkreten Ablauf.

    Lösung

    Bewertungskriterien:

    1. Use-Case: Akteure: Patient, Arzt, Sprechstundenhilfe. AFs: Termin buchen/absagen/verschieben/bestätigen/einsehen, Patient anlegen. Login als <> bei Aktionen sinnvoll.

    2. Klassen: Patient (name, geb, krankenkasse, patientennr), Termin (datum, uhrzeit, status), Arzt (name, fachrichtung). Beziehungen: Patient 1:n Termin, Arzt 1:n Termin. Methoden: Termin.bestätigen(), Patient.anlegen().

    3. Sequenzdiagramm: Patient → System: Termin anfragen. System → DB: Verfügbarkeit prüfen. DB → System: Ergebnis. System → Patient: Bestätigung/Alternativzeiten. System → E-Mail-Server: Bestätigungsmail senden.

    4. Use-Case-Diagramm – zeigt dem Auftraggeber auf einem Blick, wer das System nutzt und was es können soll. Kein technisches Vorwissen nötig.
